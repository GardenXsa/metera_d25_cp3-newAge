# AI Assistant Project Rules — Хроники Метерии

Этот файл нужен, чтобы в любом новом чате быстро восстановить правила работы над проектом.

## 1. Главная цель проекта

Мы переводим проект на более data-driven архитектуру:

- убираем захардкоженные ID, числа, пути, лимиты и правила из JS/Electron/C++ кода;
- переносим их в `data/*.json`;
- подключаем новые data-файлы через `data/runtime_manifest.json`;
- сохраняем fallback-значения в коде, чтобы проект не падал при повреждённом/отсутствующем конфиге;
- двигаемся средними/крупными subsystem-патчами по активной архитектуре, а не по мёртвым prototype-файлам.

## 2. Формат ответа

Основной формат ответа для правок проекта — JSON-патч для AI Patcher Pro:

```json
{
  "patch_name": "name",
  "operations": [],
  "commands": []
}
```

Не надо давать длинные рассуждения вместо патча, если пользователь просит продолжить работу над кодом.

## 3. Реестр работы обязателен

После каждого значимого патча нужно обновлять:

- `docs/AI_PATCHER_WORKLOG.md`

В реестре фиксировать:

1. имя патча;
2. статус;
3. добавленные/изменённые файлы;
4. что именно сделано;
5. проверки;
6. результат проверок;
7. риски;
8. следующий шаг.

## 4. Информировать пользователя

Не работать молча.

Перед крупным или рискованным патчем кратко сообщать:

- что именно делаем;
- зачем;
- какая зона риска;
- почему патч маленький/безопасный.

После логов пользователя обязательно кратко резюмировать результат.

## 5. Git-правило

После успешной пачки изменений и зелёных проверок нужно напоминать пользователю сделать commit и push.

Причина: актуальная информация для следующих чатов берётся из GitHub-репозитория.

Рекомендуемая команда после зелёного этапа:

```bash
git status
git add .
git commit -m "chore: update data-driven runtime tooling"
git push
```

Если изменения большие, лучше предложить отдельное имя commit под смысл этапа.

Важно: ассистент сам не должен утверждать, что сделал push, если пользователь не запускал команды. Нужно просить пользователя выполнить команды или дать патч/инструкцию.

## 6. Перед продолжением в новом чате

Если работа продолжается в новом чате:

1. сначала ориентироваться по GitHub-репозиторию пользователя;
2. открыть/учесть `docs/AI_PATCHER_WORKLOG.md`;
3. открыть/учесть этот файл `docs/AI_ASSISTANT_PROJECT_RULES.md`;
4. не полагаться только на старый `project_scan.txt`, потому что он может быть неактуален;
5. перед новым патчем запускать или просить запустить smoke-check.

## 7. Smoke-check

Перед крупными правками и после них использовать:

```bash
node tools/runtime_smoke_check.js
```

Или на Windows:

```bat
tools\run_runtime_smoke_check.bat
```

Зелёная базовая точка на момент создания этого файла:

```text
Summary: 50 checks, 0 failed, 0 warnings
```

## 8. Просмотр реестра

Для удобного просмотра реестра использовать:

```bat
tools\open_worklog_viewer.bat
```


Viewer также показывает прогресс data-driven переноса из:

```text
docs/DATA_DRIVEN_MIGRATION_PLAN.md
```

После каждого закрытого этапа нужно обновлять чекбоксы в этом плане, иначе прогресс-бар будет врать.


Постоянное правило: после каждого значимого патча нужно обновлять оба файла:

- `docs/AI_PATCHER_WORKLOG.md` — что сделали, проверки, результат, риски, следующий шаг;
- `docs/DATA_DRIVEN_MIGRATION_PLAN.md` — чекбоксы прогресса, текущая фаза и последняя зелёная точка.

Если обновить только код, но не эти два файла, прогресс проекта считается незафиксированным.


Также после каждого зелёного этапа нужно обновлять раздел `Ближайшие следующие шаги` в `docs/DATA_DRIVEN_MIGRATION_PLAN.md`. Пользователь должен видеть не только общий процент, но и конкретный следующий рабочий блок.

## 9. Не плодить лишние скрипты

Пользователь явно просил не решать всё через Python-скрипты.

Допустимо добавлять инструменты только если они реально помогают проекту и пользователь согласен. Предпочтение:

- JSON-патчи;
- маленькие ручные изменения;
- понятные проверки через `node --check`, JSON parse и smoke-check.

## 10. Текущая стратегия

Двигаться так:

1. работать от актуального GitHub/master и логов AI Patcher;
2. перед выбором цели проверять, что файл/подсистема активны через references, entrypoints, package/build paths и архитектурные заметки пользователя;
3. не мигрировать ProtoSystem/старые prototype-файлы только потому, что они лежат в repo;
4. делать средние/крупные subsystem-патчи по реальной архитектуре;
5. главный текущий фокус — полная поддержка моддинга: total conversion, base-data-off, manifest/merge policies, отсутствие hardcoded gameplay systems;
6. после зелёного subsystem-патча обновлять worklog/plan и делать Git checkpoint.



## 11. Размер патчей и ощущение прогресса

Пользователь отметил, что слишком маленькие шаги делают прогресс почти незаметным.

Новое правило:

- не дробить перенос на микропатчи по одной константе, если уже есть зелёная база и валидаторы;
- предпочитать средние subsystem-патчи: 3–6 связанных переносов в одной зоне;
- не смешивать разные подсистемы в одном патче;
- после среднего патча обязательно запускать smoke-check и обновлять `AI_PATCHER_WORKLOG.md` + `DATA_DRIVEN_MIGRATION_PLAN.md`;
- command-only патчи снова разрешены: AI Patcher Pro обновлён и теперь запускает команды даже без file operations.



## 12. Исправление частично зелёных патчей

Если AI Patcher Pro сообщает, что большинство операций зелёные, но 1–2 операции не найдены, не нужно пересылать весь большой патч заново.

Рабочий порядок:

1. пользователь может удалить неверные блоки в UI AI Patcher Pro;
2. ассистент должен дать маленький focused JSON patch только с исправленными операциями;
3. после применения исправленных блоков отдельно запускаются проверки/commands;
4. worklog фиксирует, что большой патч применён успешно после corrected blocks.

Это снижает шум и ускоряет работу с большими subsystem-патчами.



## 13. Сверка актуального состояния перед крупными patch

Перед каждым средним/крупным patch ассистент обязан сверяться с актуальным состоянием проекта:

1. учитывать последний GitHub/master после push;
2. учитывать последний AI Patcher log;
3. учитывать локальный `git status --short`, если он есть в логе;
4. не строить patch по старым пунктам плана, если audit показал другую активную архитектуру;
5. для рискованных replace-блоков использовать точные search-фрагменты из актуального файла.
