  Архитектурное решение:

  Структура relationships:
  npc.relationships = {
      player: {
          affection: 0,      // -100 до 100 (любовь/ненависть)
          attraction: 0,     // 0-100 (сексуальное влечение)
          trust: 0,          // 0-100 (доверие)
          intimacy: 0,       // 0-100 (близость, растёт после секса)
          sexualHistory: []  // [{ day, location, type, timestamp }]
      }
  };

  Команда updateRelationship:
  - Принимает: npcId, stat (affection/attraction/trust/intimacy), change (число)
  - Обновляет значение с учётом границ (-100 до 100 для affection, 0-100 для
  остальных)
  - Автоматически создаёт структуру relationships, если её нет

  Команда recordIntimacy:
  - Принимает: npcId, type (consensual/forced/seduction), location
  - Добавляет запись в sexualHistory с текущим днём и timestamp
  - Автоматически увеличивает intimacy на +20
  - Обновляет UI

  UI для отображения:
  - Модальное окно при клике на NPC в environment panel
  - Вкладка "Отношения" с прогресс-барами для affection, attraction, trust,
  intimacy
  - Список сексуальной истории с датами и локациями
  - Цветовая индикация (зелёный = положительное, красный = отрицательное)

  Промпт для GM:
  - Чёткие пороговые значения для консенсуального секса
  - Обязательные команды после эротической сцены
  - Модификаторы для порно-режима (снижение требований в 2 раза)

  Почему не упрощённый вариант:
  - 4 отдельных параметра вместо одного "отношение" — детализация и гибкость
  - История сексуальных сцен — возможность отслеживать развитие отношений
  - Автоматическое создание структуры — защита от ошибок GM
  - Интеграция с существующей системой NPC — консистентность

  Реестр рисков:

  - ⚠️ GM может забыть обновить отношения после сцены → Решение: Императивные
  формулировки "ты ОБЯЗАН" в промпте
  - ⚠️ Структура relationships может не создаться для старых NPC → Решение:
  Автоматическое создание при первом обращении
  - ⚠️ UI может не обновиться после изменения → Решение: Явный вызов
  updateEnvironmentVisibility() после каждой команды
